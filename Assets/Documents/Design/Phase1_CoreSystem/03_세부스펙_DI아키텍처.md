# 세부 스펙: DI 아키텍처

## 1. 목적
- 싱글톤 남용 방지
- 테스트 용이성 확보
- 모듈 간 결합도 감소

## 2. 핵심 원칙
1. 매니저급 클래스는 인터페이스로 참조
2. 직접 new/instance 호출 지양
3. 생성자는 의존성 주입만 사용
4. ScriptableObject 는 데이터 전용

## 3. DI 컨테이너

### 3.1 GameContainer
```csharp
public class GameContainer : MonoBehaviour
{
    private Dictionary<Type, object> _instances = new();
    
    public void Register<TInterface, TImplementation>(TImplementation instance)
        where TImplementation : TInterface;
    
    public T Resolve<T>();
    
    public bool IsRegistered<T>();
}
```

### 3.2 Lifetime
- Singleton: 게임 전체 수명
- Scoped: 씬 단위
- Transient: 매번 새로 생성

## 4. 등록 클래스

### 4.1 Core Services
- `IDataService<T>` → `ExcelDataService<T>`
- `IExcelConverter` → `ExcelConverter`
- `IGameManager` → `GameManager`
- `ISoundManager` → `SoundManager`

### 4.2 Gameplay Services
- `ITouchService` → `TouchService`
- `IScoreService` → `ScoreService`
- `ICharacterService` → `CharacterService`

### 4.3 UI Services
- `IUIService` → `UIService`
- `ISettingService` → `SettingService`

## 5. 사용 예시

### 5.1 등록 (GameContainer 초기화)
```csharp
var container = new GameContainer();
container.Register<IDataService<CharacterData>, ExcelDataService<CharacterData>>(
    new ExcelDataService<CharacterData>());
container.Register<IGameManager, GameManager>(
    new GameManager(container.Resolve<IDataService<CharacterData>>()));
```

### 5.2 사용 (클래스 내부)
```csharp
public class CharacterController : MonoBehaviour
{
    private readonly ICharacterService _characterService;
    
    public CharacterController(ICharacterService characterService)
    {
        _characterService = characterService;
    }
}
```
